Intro

In order to work as a team, version control is essential. Even if not working on a team, it's an important way to save stable versions of your work that can help when tracking down changes that break code later.

In particular we'll take a look at the most popular program for version control in web development: git.

The Workflow

In broad strokes the general work flow of Git is this: 1. Create your repository (start keeping track of file history) 2. Make changes and commit them to a snapshot. 3.

The first step to using git is to initialize a repository. This can be done by typing git init into the command line from your project's directory.

From there, git works by keeping track of new files that have been created or modified.

Once you've reached a point where you want to take a snapshot of your work (i.e: after fixing a bug), you can save those changes with a summary that lets others, or your future self, know what work was done.

You have to manually specify which files you want to include in the snapshot, which is discussed more under Commits.

From there that change can be shared with your team so they can merge that work into their own copies. The reverse is also possible, you can merge fixes others have made.
